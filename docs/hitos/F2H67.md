## Qué entregó

Vehicle physics arcade-ish con `JPH::VehicleConstraint` + `WheeledVehicleController` nativos de Jolt. Chassis + 4 wheels independientes que rotan visualmente (vía sub-mesh selector nuevo), mount/dismount con tecla F, chase cam orbital third-person, Lua API para input desde scripts, asset `.moodvehicle` JSON con tuning persistible, Inspector dedicado, debug overlay 3D bajo F1, modelo CC0 Kenney Car Kit + sample mapa + demo project precargable.

### Bloque A — `VehicleConfig` puro

[`src/engine/physics/vehicle/VehicleConfig.{h,cpp}`](../../src/engine/physics/vehicle/VehicleConfig.h). Header sin Jolt, testeable headless. Struct declarativa: `WheelConfig` (attachLocal, radius, width, suspension, friction, driven/steered/handbraked), `EngineConfig` (maxTorque, RPM range, gear ratios + reverse, brake/handbrake torque), `VehicleConfig` (chassis halfExtents/mass/CoM, 4 wheels, engine, steer angle/lerp).

**Defaults SA en `makeDefaultSA()`**: chassis 1500 kg con CoM bajo (Y -0.20m → no flipea fácil), wheels 0.35m radius con friction lateral 1.4 + longitudinal 1.6 (alta tracción), motor 500 Nm @ 4000 RPM, 5 marchas + 1 reversa, brake 1500 Nm, handbrake 4000 Nm, steering 35° con lerp 4.0/s (responsivo arcade). 4WD por default (delanteras steered, traseras handbraked).

Validación: `vehicle::isValid(cfg)` chequea NaN/inf, masas > 0, gears decrecientes, steerAngle <= 90°, etc.

### Bloque B — Componentes + persistencia

`VehicleComponent` + `VehicleSeatComponent` en [`Components.h`](../../src/engine/scene/components/Components.h). Input state (throttle/brake/steer/handbrake) que Lua o el Player escriben cada frame. Runtime fields (`vehicleId`, `wheelEntities[4]`) NO persistidos. `configPath` referencia a `.moodvehicle`. Persistencia aditiva en `.moodmap` (sin schema bump): mapas pre-F2H67 cargan igual.

Round-trip cubierto por 2 tests headless en [`test_scene_serializer_lighting_physics.cpp`](../../tests/test_scene_serializer_lighting_physics.cpp).

### Bloque C — `PhysicsWorld_Vehicle.cpp` wrapper Jolt

[Source](../../src/engine/physics/world/PhysicsWorld_Vehicle.cpp). API expuesta:

```cpp
u32  createVehicle(const VehicleConfig&, const glm::mat4& initialWorldTransform);
void destroyVehicle(u32);
void setVehicleInput(u32, f32 throttle, f32 brake, f32 steer, f32 handbrake);
bool readVehicleState(u32, VehicleState& out) const;
void applyVehicleImpulse(u32, const glm::vec3&);
u32  vehicleChassisBodyId(u32) const;
u32  vehicleCount() const;
```

`VehicleState` retorna chassis world matrix + 4 wheel world matrices (computed via `VehicleConstraint::GetWheelLocalTransform`) + forwardSpeed (proyección de velocidad sobre +Z local) + grounded (al menos 1 wheel toca) + currentGear + engineRPM.

Internals:
- Chassis = `JPH::BoxShape` (con `OffsetCenterOfMassShape` wrapper si CoM != 0) + `JPH::BodyCreationSettings` Dynamic.
- `VehicleConstraintSettings.mUp = Y+`, `mForward = Z+`, `mMaxPitchRollAngle = 60°`.
- 4 `WheelSettingsWV` con suspension `FrequencyAndDamping` mode, friction curves multi-point (SA-style flat-ish que no decae con slip).
- 1-2 `VehicleDifferentialSettings` según wheel.driven flags (4WD = 2 diffs front+rear con `engineTorqueRatio=0.5`; 2WD = 1 diff con ratio 1.0).
- Engine `mNormalizedTorque` curve con pico en `maxTorqueRPM/maxRPM`.
- Transmission `Auto` mode con shiftUp 75% RPM y shiftDown 30%.
- `VehicleCollisionTesterRay` (ObjectLayer::Moving) — más estable que shape cast para arcade.
- Registro como `StepListener` del physics system.

7 tests runtime headless en [`test_physics_vehicle.cpp`](../../tests/test_physics_vehicle.cpp): createVehicle handle válido, config inválido → 0, destroy idempotente, settle on ground, throttle → forwardSpeed > 5 m/s tras 3s, brake decrece speed, applyImpulse mueve chassis, multi-vehicle independientes.

### Bloque D — `VehicleSystem`

[Source](../../src/systems/physics/VehicleSystem.cpp). Hookeado en `EditorScene::tickPhysics` después del step + después del RagdollSystem. Loop por `<VehicleComponent, TransformComponent>`:

1. **Materialize lazy** si `dirty && vehicleId == 0`: resuelve `VehicleConfig` (asset loader o makeDefaultSA fallback), llama `createVehicle`, busca las 4 child-entities por tag fijo (`Wheel_FL/FR/RL/RR`) y cachea sus handles en `wheelEntities[]`. Si alguna falta, log warn pero sigue (physics OK sin sync visual de esa wheel).
2. **Push input**: `setVehicleInput(...)` con los valores del componente.
3. **Sync pose**: `readVehicleState` → escribe Transform del chassis entity + cada wheel child-entity (extracción Euler XYZ vía `glm::extractEulerAngleXYZ`).

### Bloque E — Lua bindings

[`LuaBindings_Vehicle.cpp`](../../src/engine/scripting/bindings/LuaBindings_Vehicle.cpp). Tabla `vehicle` con 6 funciones, todas lookup por tag (primer match gana, mismo patrón que `ragdoll.*` y `inventory.*`):

```lua
vehicle.set_input("Banshee", {throttle=0.7, brake=0, steer=-0.3, handbrake=0})
vehicle.get_speed("Banshee") -> m/s
vehicle.is_grounded("Banshee") -> bool
vehicle.gear("Banshee") -> int  (-1 reverse, 0 neutral, 1..N)
vehicle.rpm("Banshee") -> float
vehicle.respawn("Banshee", {x=0, y=1, z=0}, rotYDeg)
```

`respawn` destruye + recrea el vehicle con `dirty=true`. Reset de input + teleport del Transform.

### Bloque F — Player ↔ Vehicle mount/dismount + chase cam

[`EditorPlayMode.cpp::updatePlayer`](../../src/editor/application/EditorPlayMode.cpp) extendido. Estado nuevo en `EditorApplication.h`: `m_playerMountedVehicleEntity` (handle raw, 0 = on foot), `m_fKeyPrevFrame`, `m_chaseDistance` (5m default), `m_chaseHeightOffset` (1.5m).

**Mount**: F edge-press cuando on-foot → busca entities con `VehicleComponent` dentro de radio 3m del player char. Primer match gana → set `playerMountedVehicleEntity = handle`. SDL relative mouse mode ON (para chase cam).

**Dismount**: F edge-press cuando mounted → reset input (brake=1, handbrake=1 → auto se detiene), teleport del char a `chassis.position + (1.5, 1, 0)` (costado derecho). `playerMountedVehicleEntity = 0`.

**Input dispatch when mounted**: WASD escribe directamente al `VehicleComponent.input*` (no pasa por Lua — más responsivo). Space = handbrake. Char controller queda con `setCharacterMovement({0,0,0})`.

**Chase cam**: `m_playCamera.applyMouseMove(mx, my)` actualiza yaw/pitch normal. Posición re-derivada cada frame como `chassisPos - cam.forward() * m_chaseDistance + (0, m_chaseHeightOffset, 0)`. El FpsCamera ya tiene `forward()` derivado de yaw/pitch → al teleportar position y dejar la orientación, naturalmente mira hacia el chassis sin necesidad de lookAt. Mouse rota el orbit; sin input, sigue la cam donde quede.

**Gate adicional en `EditorScene.cpp`**: el sync post-step cam→char solo corre si `playerMountedVehicleEntity == 0` — sino la chase cam quedaba pisada cada frame.

### Polish 1 — Sub-mesh selector en `MeshRendererComponent`

Añadido `std::string subMeshName` a [`MeshRendererComponent`](../../src/engine/scene/components/Components.h) y `std::string name` a [`SubMesh`](../../src/engine/render/resources/MeshAsset.h). `MeshLoader.cpp` puebla `name` desde `aiMesh->mName`.

Render path en [`SceneRenderer_Render.cpp`](../../src/engine/render/scene_renderer/SceneRenderer_Render.cpp): si `mr.subMeshName != ""`, skipea sub-meshes cuyo `sub.name != mr.subMeshName`. RenderBatching excluye entities con sub-mesh filter del path instanced (van al `drawMeshRenderer` no-batched — OK porque son pocas: chassis + 4 wheels por auto).

Esto permite que **5 entities distintas compartan UN solo MeshAsset** (`sedan.fbx`): cada una renderiza solo su sub-mesh, las wheels rotan independientes con el chassis.

## Bugs cazados durante implementación + fixes

1. **`GLM_GTX_euler_angles` requiere `#define GLM_ENABLE_EXPERIMENTAL`** — el include falla sin él. Fix en VehicleSystem.cpp.
2. **PDB lock por `-m:8`** — paralelismo MSVC alto hace que varios CL.EXE escriban al mismo vc143.pdb. Workaround: `-m:4` o `/FS`. No es F2H67-specific.
3. **ABI mismatch tras agregar `std::string name` a `SubMesh`** — primer build mostró crash en `createMaterialsForMesh` por `mood_engine_lib.lib` y tests linkeando contra layouts distintos de SubMesh. Clean rebuild lo resolvió. CMake header dependency tracking no detectó el cambio en MeshAsset.h.
4. **Gate del SceneSerializer filtraba entities con solo Tag+Transform** — los test fixtures de F2H67 con solo `VehicleComponent` no se persistían. Fix: extender gate con `hasVehicle` + reconocer tags `Wheel_FL/FR/RL/RR` para wheels-placeholder.
5. **`unique_ptr<vehicle::VehicleConfig>` con forward decl en AssetManager.h** — el destructor out-of-line de AssetManager necesita la definición completa de VehicleConfig. Fix: include en AssetManager.cpp.

## Decisiones clave (detalle en `DECISIONS.md`)

1. **Backend `JPH::WheeledVehicleController`** nativo de Jolt (no rolling-our-own). Tiene engine + transmission + differentials + suspension + friction curves listos. Vehicle constraint nativo testeado.
2. **Defaults SA-style** baked en `makeDefaultSA()`: alta tracción (lateral 1.4, longitudinal 1.6), CoM bajo (-0.20m Y), motor responsivo (500 Nm, 35° steer, lerp 4/s), handbrake 4000 Nm para derrapes controlados. 4WD por default — más estable que RWD para arcade.
3. **Ensamblaje multi-entity**: 1 chassis entity (`VehicleComponent`) + 4 child entities (tags fijos `Wheel_FL/FR/RL/RR`). Permite swappear meshes de wheels independientes en el editor sin tocar physics. Auto-find por tag al primer tick.
4. **Sub-mesh selector** en `MeshRendererComponent.subMeshName` para que 5 entities compartan 1 FBX. Render path skipea sub-meshes que no coinciden + RenderBatching excluye del path instanced (drawMeshRenderer no-batched).
5. **Asset `.moodvehicle` JSON aditivo** con slot 0 = `makeDefaultSA()` lazy-init. Cada campo opcional → fallback default. Mismo patrón que Dialog/Item/Quest.
6. **Mount/dismount con F + radio 3m** (no via componente Trigger del engine). Input dispatch al vehicle component directo en C++ (más responsivo que pasar por Lua cada frame).
7. **Chase cam reusa `FpsCamera`**: yaw/pitch del mouse rotan el orbit; position se re-deriva cada frame como `chassisPos - forward() * distance + (0, height, 0)`. `forward()` apunta naturalmente al chassis sin necesidad de lookAt.
8. **Persistencia aditiva, runtime no persiste**: configPath persiste pero `vehicleId`, `wheelEntities[]`, `input*` no. `dirty=true` al cargar para que el VehicleSystem materialice.

## Limitaciones conocidas v1 (agendadas a futuros hitos)

- **Sin audio del motor** (engine sound RPM-mapped). Hito polish si emerge demanda.
- **OBB del chassis en debug overlay usa proxy fijo** `(0.9, 0.5, 2.0)`. Si el dev tunea halfExtents en su `.moodvehicle`, el overlay queda desfasado. Agendable: leer halfExtents reales del VehicleConstraint.
- **Inspector Vehicle no undoable** — cambios de `configPath` no entran al HistoryStack. Polish chico.
- **Sin damage model**: chocas y el auto no se rompe. Plan original.
- **Chase cam sin zoom** (`m_chaseDistance` fijo en 5m). Polish — scroll wheel para ajustar.
- **Sin múltiples seats** (pasajero). Un asiento por vehículo en v1.
- **NPC drivers** (AI conduciendo): integración con NavAgent. Out of scope.
- **Motorcycles / tracked vehicles**: `JPH::MotorcycleController` y `TrackedVehicleController` existen en Jolt pero out of scope F2H67.
