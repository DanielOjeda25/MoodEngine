# F2H66 — Ragdolls (auto-build desde skeleton Mixamo)

> **Tag**: `v1.53.0-fase2-hito66`
> **Fecha de cierre**: 2026-05-18
> **Sub-fase**: 2.4 — Física avanzada (continuación de F2H65). Cierra el plan original F2H24.
> **Origen**: el dev quería *"que este sea un motor físico realista, como el source"* — F2H65 entregó joints; F2H66 los compone automáticamente sobre el esqueleto de un NPC para flopping físico al morir, estilo HL2.

---

## Decisiones de scope (acordadas con el dev al iniciar el hito)

1. **Disparo manual via Lua + impulse al activar** (HL2: el cadáver vuela en la dirección del disparo). La auto-detección HP→0 vive en el script del NPC, no es scope del engine.
2. **Sin vuelta a animación** — una vez ragdoll, queda ragdoll para siempre (recargar mapa = reset). Vuelta animación↔ragdoll con IK transition agendada como follow-up si emerge necesidad.
3. **Pocos simultáneos** (1-5). Apoyamos sleeping nativo de Jolt para que los muertos en reposo no consuman CPU.
4. **Auto-build sobre Mixamo** — convención `mixamorig:*` da mapping bone→body sin config manual. Esqueletos no-Mixamo retornan layout vacío + warn (defensivo, no crash).
5. **Solo Play Mode** — no preview en Editor.

## Qué entregó

- `PhysicsWorld.cpp` 850 → **611 LOC**: bodies + step + raycast + sync de F2H40.
- `PhysicsWorld_Constraints.cpp` **152 LOC**: hinge/distance/point de F2H65 movido aquí.
- `PhysicsWorld_Ragdoll.cpp` **269 LOC**: API nueva (Bloque C).
- `PhysicsWorld_Internal.h` **125 LOC**: PIMPL compartido (Jolt headers + `JPH::PhysicsSystem*` + maps internos).

Mismo patrón que `EditorUI_*.inl`, `InspectorPanel_*.cpp`, `EditorRenderPass*.cpp` ya tenían. Build verde sin cambio funcional antes de avanzar.

### Bloque A — Layout helper (puro, sin física)

Nuevos archivos [`src/engine/physics/ragdoll/RagdollLayout.{h,cpp}`](../../src/engine/physics/ragdoll/RagdollLayout.h). Header puro (sin Jolt) — usable headless en tests.

**Convención**: 14 bodies fijos para Mixamo:

```
Hips (root) → Spine → Spine1 → Spine2 → Neck → Head
                                  ↳ LeftArm  → LeftForeArm
                                  ↳ RightArm → RightForeArm
Hips → LeftUpLeg  → LeftLeg
     → RightUpLeg → RightLeg
```

**Wrist / Foot / Shoulder / fingers NO se ragdollean** — se montan sobre el parent ragdolleado en el sync (heredan su pose).

**Constraints por par parent→child**:
- **Hinge** (1 DOF): codos (`ForeArm`) y rodillas (`Leg`).
- **SwingTwist** (3 DOF cono + twist): hombros, caderas, spine, cuello.

**Mass distribution**: cada body recibe masa proporcional al volumen del capsule (`π·r²·h + (4/3)·π·r³`), normalizado para que la suma == `totalMass` (default 70 kg).

**Cap defensivo**: si NINGÚN bone del esqueleto tiene prefijo `mixamorig:`, retorna `RagdollLayout{}` vacío + `Log::warn`. El RagdollSystem chequea `empty()` y revierte `state = Animated` sin crashear.

Tests en [`tests/test_ragdoll_layout.cpp`](../../tests/test_ragdoll_layout.cpp) (6 tests): skeleton Mixamo standard → 14 bodies + 13 constraints + masas suman 70±0.5%; codos/rodillas son Hinge; resto SwingTwist; skeleton vacío → layout empty; skeleton sin prefix → layout empty.

### Bloque B — RagdollComponent

Componente nuevo en [`Components.h`](../../src/engine/scene/components/Components.h):

```cpp
struct RagdollComponent {
    enum class State : u8 { Animated = 0, Ragdolling = 1 };
    State state = State::Animated;

    /// Config tweak persistente.
    f32  totalMass = 70.0f;
    f32  limbRadius = 0.05f;
    bool useGravity = true;

    /// Impulse al body torso al transicionar (world coords). Magnitud
    /// típica: 5-200 N·s para un NPC de 70 kg.
    glm::vec3 spawnImpulse{0.0f};

    // --- Runtime (NO persiste) ---
    u32  ragdollId = 0;     // handle del PhysicsWorld; 0 = no creado
    bool dirty = true;
};
```

Persistencia aditiva en `.moodmap` (sin schema bump). **`state` NO se persiste** — siempre arranca `Animated` al cargar (los NPCs muertos persistidos no son scope; v1 asume que el dev guarda PRE-evento si quiere preservar el momento).

### Bloque C — PhysicsWorld Ragdoll API (en `PhysicsWorld_Ragdoll.cpp`)

```cpp
u32  createRagdoll(const ragdoll::RagdollLayout& layout,
                   const std::vector<glm::mat4>& partWorldTransforms,
                   bool useGravity);
void destroyRagdoll(u32 ragdollId);
bool readRagdollPose(u32 ragdollId, std::vector<glm::mat4>& outWorlds) const;
void applyRagdollImpulse(u32 ragdollId, int partIndex, const glm::vec3& impulseWorld);
u32  ragdollCount() const;
```

Construye `JPH::RagdollSettings` desde el layout: 1 `Part` por bone con shape `CapsuleShape` + transform world inicial. Crea `HingeConstraint` o `SwingTwistConstraint` entre Parts según `ConstraintKind`. Setea `GroupFilterTable` para deshabilitar self-collision (sino el ragdoll se "explota" por contactos internos).

**`JPH::Skeleton` espejo** (en el storage interno junto al `Ref<Ragdoll>`) — necesario porque `Ragdoll::Stabilize()` requiere acceder al `Skeleton` paralelo de `mParts`. Sin él, el ragdoll funciona pero no podés llamar Stabilize.

Storage: `std::unordered_map<u32, RagdollEntry>` con `RagdollEntry { JPH::Ref<JPH::Ragdoll>, JPH::Ref<JPH::Skeleton> }` + `nextRagdollId = 1` (id == 0 == "no ragdoll").

Tests en [`tests/test_physics_ragdoll.cpp`](../../tests/test_physics_ragdoll.cpp) (7 tests): handle válido + count, layout vacío → 0, applyImpulse mueve el torso, readPose retorna size correcto, destroyRagdoll idempotente, ragdoll bajo gravedad cae sin atravezar al infinito, multi-ragdoll coexistencia.

### Bloque D — RagdollSystem (módulo dedicado)

Sistema en [`src/systems/physics/RagdollSystem.{h,cpp}`](../../src/systems/physics/RagdollSystem.h). EditorScene::tickPhysics solo invoca `RagdollSystem::tick(*m_scene, *m_assetManager, *m_physicsWorld)` — la implementación no contamina EditorScene.cpp.

Loop iterando `<RagdollComponent, AnimatorComponent, SkeletonComponent, MeshRendererComponent, TransformComponent>`:

1. **Transición `Animated → Ragdolling`** (cuando `state == Ragdolling && ragdollId == 0`):
   - Build layout desde `mesh.skeleton`.
   - Convertir skinning matrices del Animator → mesh-space globals (`global[i] = skinning[i] * bind[i]`).
   - Componer world transforms (`entityWorldMatrix * meshGlobal[i]`).
   - `createRagdoll(layout, worldXforms, useGravity)`.
   - Aplicar `spawnImpulse` al body torso (default index = 3, Spine2).
   - El `AnimationSystem` deja de pisar skinning (ver Bloque G).
2. **Sync `Ragdolling`** (cada frame post-step):
   - `readRagdollPose(ragdollId, outWorlds)`.
   - Convertir world → mesh-space → skinning matrices con `inverseBind` del Skeleton.
   - Bones no-ragdolleados (manos, pies, dedos, shoulders) heredan del parent ragdolleado más cercano.

Cleanup integrado en [`DeleteEntityCommand`](../../src/editor/commands/DeleteEntityCommand.h): callback `RagdollCleanup` análogo a `BodyCleanup`/`ConstraintCleanup` de F2H40/F2H65. EditorApplication inyecta lambda que llama `PhysicsWorld::destroyRagdoll`.

### Bloque E — Lua bindings

Nueva tabla `ragdoll` en [`LuaBindings_Ragdoll.cpp`](../../src/engine/scripting/bindings/LuaBindings_Ragdoll.cpp):

```lua
ragdoll.enable("NPC")                              -- sin impulse, cae por gravedad
ragdoll.enable("NPC", {x = 0, y = 80, z = -40})    -- + impulse atrás-arriba (HL2)
ragdoll.is_ragdolling("NPC") -> bool
```

Patrón "lookup por tag" (no por handle) consistente con `inventory.give(tag, ...)` de F2H52. Primer match gana (convención Hammer-style); si no encuentra → `log.warn` y no-op.

La función `enable` solo setea `state = Ragdolling` + `spawnImpulse` + `dirty = true`. La materialización física real (bodies + constraints) la hace `RagdollSystem` en el siguiente tick. Mismo patrón lazy que `JointComponent.dirty` de F2H65.

### Bloque F — Debug overlay 3D

En [`EditorRenderPass_Overlay.cpp::drawEditorScene3DOverlay`](../../src/editor/application/EditorRenderPass_Overlay.cpp), bajo F1 toggle, junto al overlay de joints de F2H65:

Validado en runtime: el dev abrió el demo, activó F1, confirmó que las capsules se superponen al mesh y que las líneas amarillas conectan los joints correctamente. Sin issues anatómicos.

### Bloque G — Polish: bug de race AnimationSystem ↔ RagdollSystem

**Bug descubierto en validación** (post-Bloque E): el ragdoll se creaba correctamente en Jolt (log `createRagdoll: id=1 parts=14` OK), el trigger Lua llamaba `ragdoll.enable("NPC", ...)`, el RagdollSystem escribía las skinning matrices del ragdoll en `skel.skinningMatrices`... pero **el NPC seguía visualmente en pose idle**, "ni se inmutaba".

**Causa**: el frame order es `AnimationSystem → tickPhysics (= step + RagdollSystem) → render`. El `AnimationSystem::update` siempre pisaba `skel.skinningMatrices` con la pose del Animator (incluso con `playing = false`, porque "no avanzar el tiempo" no implica "no escribir matrices" — la pose congelada en `t = último frame` se sigue evaluando cada frame).

**Fix** en [`AnimationSystem.cpp:75-87`](../../src/systems/animation/AnimationSystem.cpp): guard al inicio del lambda — si la entity tiene `RagdollComponent` en estado `Ragdolling`, return inmediato (no tocamos sus skinning matrices). El `RagdollSystem` es el único escritor mientras dure el ragdoll.

```cpp
if (e.hasComponent<RagdollComponent>() &&
    e.getComponent<RagdollComponent>().state
        == RagdollComponent::State::Ragdolling) {
    return;
}
```

## Bugs cazados durante implementación + fixes

1. **`LuaBindings_Ragdoll.cpp` faltaba en `tests/CMakeLists.txt`** — el binding se agregó al executable principal en Bloque E pero el header estaba referenciado desde tests vía `LuaBindings::registerAll`. Linker error `setupRagdollBindings unresolved` al correr el test suite. Fix: línea nueva en `tests/CMakeLists.txt:192`.
2. **Sample `.moodproj` con `version: 3` no-existente** — al crear el demo project en otra máquina, el primer JSON tenía `version: 3` (typo). `k_MoodprojFormatVersion = 1`. El loader rechazaba con "unsupported version". Fix: `"version": 1`.
3. **imnodes dep no resolved en build/debug** — el dev volvió al repo en una máquina con la build cacheada pre-F2H62 (que es cuando se agregó imnodes via CPM). `cmake --preset windows-msvc-debug` fetcheó el dep y resolvió.
4. **Race AnimationSystem vs RagdollSystem** — el bug central de Bloque G (ver arriba). Sin este fix, el ragdoll se creaba en Jolt pero el mesh visual seguía en idle. Descubierto solo en runtime — los tests headless no ejercitan el frame loop completo.

## Decisiones clave (detalle en `DECISIONS.md`)

1. **Backend `JPH::Ragdoll` nativo de Jolt**, no rolling our own desde joints sueltos. Jolt expone `RagdollSettings` declarativo + `AddToPhysicsSystem` + `GetBodyState` + `Stabilize`. Reinventarlo desde `HingeConstraint`/`SwingTwistConstraint` sueltos sería trabajo redundante.
2. **Auto-build sobre convención Mixamo** (prefijo `mixamorig:*`) — esqueletos custom retornan layout vacío + warn (defensivo, no crash). Per-mesh ragdoll config (no-Mixamo skeletons) queda agendada como follow-up.
3. **Bodies = capsules, NO boxes**. Mejor llenado del cuerpo humano + performance OK con N=14. Override per-bone agendable si emerge presión.
4. **14 bones, no 22+**. Wrist/Foot/Shoulder/fingers se montan sobre el parent ragdolleado (heredan pose en el sync). Reduce constraints + simulación; no se nota visualmente para NPCs distantes.
5. **Mass distribution proporcional al volumen** (`π·r²·h + (4/3)·π·r³`) normalizado a `totalMass`. Más realista que `totalMass / N`.
6. **Sin vuelta animación↔ragdoll** en v1 (convención HL2). El return-from-ragdoll requiere IK transition + pose matching, scope de un hito propio.
7. **Disparo via Lua `ragdoll.enable(tag, {x,y,z})`** — no via componente Trigger del engine. Mantiene el control en el gameplay layer. Mismo patrón que `inventory.give(tag, ...)` de F2H52.
8. **Frame-order fix**: `AnimationSystem` skipea entities ragdolleando. El `RagdollSystem` es el único escritor de skinning matrices mientras dure el ragdoll. Sin esto, el sync queda pisado cada frame.
9. **Persistencia aditiva (sin schema bump)**: `RagdollComponent` opcional en `.moodmap` v14. `state` NO persiste (siempre arranca `Animated`). Mapas pre-F2H66 cargan igual.
10. **Split `PhysicsWorld.cpp`** en 4 archivos (core / Constraints / Ragdoll / Internal PIMPL) antes de meter API nueva. Patrón ya usado en `EditorUI_*.inl`, `InspectorPanel_*.cpp`.

## Limitaciones conocidas v1 (agendadas a futuros hitos)

- **Vuelta animación ↔ ragdoll** (`disableRagdoll` + IK transition). Necesita pose matching desde la pose del ragdoll hasta una keyframe del Animator. Diferido.
- **Partial ragdoll** (golpe al brazo lo dobla, NPC sigue vivo). Plan original F.E.A.R.-style. Diferido.
- **Ragdoll persistente en `.moodmap`** (guardar el estado mid-flop). Diferido — pocos casos reales.
- **Hit reactions** (impulso a un body específico sin matar al NPC). Out of scope; encajaría en partial ragdoll.
- **Per-mesh ragdoll config** (esqueletos no-Mixamo: cuadrúpedos, criaturas custom, Fox.glb). Out of scope F2H66.
- **Tuning de masses y limits desde Inspector**. v1 expone solo `totalMass`/`limbRadius`/`useGravity`/`spawnImpulse` en la struct; el Inspector aún no los pinta. Polish chico, agendable.
- **NPC sin collider estático en estado Animated**: el player atraviesa al NPC hasta que el ragdoll arranca. Esperado para el demo; un NPC "vivo" debe tener un `RigidBodyComponent` con `BodyType::Static` o `Kinematic` que se destruya al transicionar a ragdoll. No-scope F2H66.
