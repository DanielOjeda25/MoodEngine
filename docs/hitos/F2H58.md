Bloques A-J + 1 fix lateral, tercer hito de **Sub-fase 2.6 — Render polish** (bloom F2H55 → SSAO F2H56 → **color grading F2H58** → god rays/F2H6X+). **Origen**: post-cierre F2H57 (pivot UX), retomamos el plan de render polish. Pre-arranque el dev preguntó cómo organizan otros motores los settings de post-process; la respuesta — un solo panel consolidado tipo Unity Volume / Unreal Post Process Volume / Godot WorldEnvironment — se convirtió en Bloque A (refactor de la UI). Bloques B-E entregan el algoritmo de color grading; Bloques G-I son UX polish surgido del primer tour visual del dev (reset buttons, preset dropdown, jerarquía Unreal-style); Bloque F descubre y arregla un bug pre-existente del MenuBar; Bloque J cierre. **Bloque A — Consolidación del panel Environment** (commit `9ea434a`): extracción de `renderEnvironmentSection` de `InspectorPanel_Light.cpp` a un archivo propio `src/editor/panels/scene/InspectorPanel_Environment.cpp` (~222 líneas). El dispatch en `InspectorPanel.cpp` ya separaba `LightComponent` de `EnvironmentComponent`, así que el cambio real es de organización: cada sub-bloque (Sky+Fog / Tonemap+Exposición / Bloom / SSAO) ahora es un `CollapsingHeader` independiente con `ImGuiTreeNodeFlags_DefaultOpen`. Comportamiento + lógica + undo de cada widget preservados literalmente. **Bloque B — Shader + pase de Color Grading**: `shaders/color_grading.frag` nuevo con algoritmo Unity URP standard — LUT 2D 256x16 layout horizontal (16 slices de 16x16 px representando el cubo RGB 16x16x16), interpolación lineal entre 2 slices adyacentes para componente Blue intermedio, `mix(originalColor, gradedColor, intensity)` para blend. Pre-tonemap (opera en LDR-ish range, después de bloom+SSAO pero antes de la curva tonemap). Reusa `post_process.vert` para el fullscreen triangle. Clase C++ `src/systems/render/ColorGradingPass.{h,cpp}` (~120 líneas) sigue el mismo patrón de `BloomPass`/`SSAOPass`: `apply(src, dst, lutTextureId, intensity) → bool` con save/restore GL state. **Bloque C — Wireup en `SceneRenderer` + extensión `EnvironmentComponent`**: 3 campos nuevos (`colorGradingEnabled` default OFF — a diferencia de bloom/SSAO, sin LUT no aporta y look default cuestionable; `colorGradingLutPath` vacío = identidad; `colorGradingIntensity` 1.0 default). `SavedEnvironment` paralelo en `SceneSerializer.h`. `EntitySerializer` write aditivo (solo persiste si difiere del default — patrón establecido en bloom/SSAO). `SceneLoader` copia Saved → Component. `SceneRenderer` nuevos members: `m_colorGradingPass`, `m_colorGradingFb` (HDR intermedio), `m_currentLutTextureId`, `m_identityLutId`. `synthesizeIdentityLut()` genera 256x16 RGBA8 lineal en runtime (formula `pixel(x,y) = ((x%16)*255/15, y*255/15, (x/16)*255/15, 255)`) lazy en primer uso — backup garantizado si el LUT path no resuelve. `endFrame()` inserta el pase entre bloom y post-process: `if (colorGradingEnabled && lutResolved && apply()) postProcessSrc = colorGradingFb; else postProcessSrc = afterBloom`. Resolución LUT desde path usa el existente `AssetManager::loadTexture` como cache; si falla, fallback a identidad. **Bloque D — UI sliders iniciales** (primera versión del Color Grading section en `InspectorPanel_Environment.cpp`): checkbox `activar color grading` + path field con `InputText` + botón `...` para file picker filtrado a PNG + slider `intensidad` (0..1) + botón "LUT identidad (reset)". Undo via `pushEditIfDone` al soltar el slider. 9 i18n keys nuevas (`color_grading*`). **Bloque E — 4 LUTs sample shipados** en `assets/luts/`: `identity.png` (testing del algoritmo), `cinema_warm.png` (atardecer / sepia / westerns), `matrix_cool.png` (verde-cyan sci-fi), `noir_high_contrast.png` (B&N con S-curve smoothstep). Generados vía `tools/gen_luts.py` (Python + PIL) — el dev del juego puede sumar `color_func(r,g,b) → (r,g,b)` y regenerar. `assets/luts/README.md` documenta el formato + workflow alternativo (DaVinci Resolve / Photoshop / Filmlight Baselight). **Bloque F — validación visual + fix MenuBar pre-existente**: dev lanzó el editor → crash con assert ImGui `EndMenuBar` line 9198 (`window->Flags & ImGuiWindowFlags_MenuBar` falló). **Root cause**: F2H57 Bloque E (commit `3a9a072`) eliminó las ~15 entries del submenu `Ayuda > Demos` pero **borró dos `EndMenu()` por error** — uno del submenu interno (correcto eliminar) y otro del menu `Help` externo (no debía irse). Esto dejaba el Help menu como un `BeginMenu` sin `EndMenu`, rompiendo el balance del stack de ImGui y disparando el assert en el siguiente frame con BeginMenuBar. Fix: una sola línea `ImGui::EndMenu();` agregada al final del `if (BeginMenu("Help"))` en `MenuBar.cpp:256`. **Bloque G — reset-to-default por sección**: al primer tour del dev en Color Grading, pidió botones de "restablecer" estilo Unity Inspector / Unreal Details. Helper `drawSectionResetButton(idSuffix, applyFn)` que pinta un `SmallButton` alineado a la derecha con icon `ICON_FA_ROTATE` + tooltip; al click, asigna los campos de esa sección desde una instancia `kEnvDefaults` (constexpr default-constructed `EnvironmentComponent`). Bug detectado durante el primer tour: los 5 botones tenían el mismo label visible "Restablecer" → mismo ImGui ID interno → solo uno respondía al click. Fix: el helper toma un `idSuffix` (`fog`/`tonemap`/`bloom`/`ssao`/`cgrade`) que se concatena al label como `##envreset_<id>` para forzar IDs únicos sin cambiar el texto visible. Sin undo en v1 (snapshot multi-campo requeriría `ResetSectionCommand` custom — diferido a follow-up si emerge demanda). **Bloque H — LUT dropdown de presets**: el dev pidió: *"lo de LUT no lo veo viable, a menos que tenga pressets incluidos"*. Reemplazo del `InputText` + botón `...` + "LUT identidad reset" por un `BeginCombo` con 5 built-ins hardcodeados (Ninguno / Cálido / Frío / Noir / Identidad-testing) + opción "Personalizado..." al final que dispara el file picker existente. El path interno sigue siendo `colorGradingLutPath` — el preset solo es UX sintáctica encima. Si el path actual no matchea ningún built-in, el combo muestra "Personalizado: \<filename\>". 8 i18n keys nuevas (`color_grading_preset*`). **Caracter Unicode `…` (U+2026)** detectado renderizando como `?????` en la font default de ImGui — reemplazado por `...` ASCII (la font cargada no incluía la página Unicode general punctuation). **Bloque I — reorganización jerárquica Post-Process**: top-level del Inspector→Environment ahora es solo 2 headers (`Sky + Fog` y `Post-procesado`); el wrapper `Post-procesado` contiene los 4 sub-pases (Tonemap+Exposición / Bloom / SSAO / Color Grading) como CollapsingHeader anidados, separados con `ImGui::Indent()/Unindent()` para indicar subordinación visual. Convención Unreal Post Process Volume / Unity Volume. Nueva i18n key `tonemap_section` para el sub-header (la key existente `post_process` queda asociada al wrapper). **Bloque J — cierre**. **Limitaciones conocidas v1 documentadas**: (a) color grading aplicado pre-tonemap en HDR clamped a [0,1] — convención Unity URP. Si emerge que las LUTs de DaVinci Resolve (que asumen input post-tonemap) no quedan bien, evaluar switch a post-tonemap. (b) Sin soporte de `.cube` (Adobe ASCII format) — solo PNG 256x16. (c) Sin scan dinámico del directorio `assets/luts/` para el dropdown — los 5 built-ins son hardcodeados; LUTs adicionales se accedan vía "Personalizado...". (d) sRGB vs linear loading del LUT: el `AssetManager::loadTexture` carga texturas con el flag estándar (probablemente sRGB); la LUT es una tabla de mapeo, no una imagen para mirar, así que debería cargarse linear — limitación tolerable mientras los 4 built-ins sean PNGs lineales generados por nuestro script. (e) Reset de sección sin undo. **NO entra en F2H58**: god rays / shafts volumétricos (F2H59 estaba previsto pero hito siguiente reasignado a primitivas geométricas según pedido del dev), shadow polish (CSM / contact shadows), preview 3D del LUT (slot extra en el panel para visualizar el cubo RGB del LUT cargado), `.cube` parser, scan dinámico del directorio LUTs, command para undo del reset multi-campo. **Decisiones**: (1) **Pre-tonemap LUT (convención Unity URP)** sobre post-tonemap (Unreal): URP es más simple porque opera siempre en mismo color space; switch a Unreal-style si emerge feedback de coloristas. (2) **Default OFF** (a diferencia de bloom/SSAO): sin LUT el efecto no aporta nada, y prender por default un look específico es decisión de art direction que el motor no debe imponer. (3) **Path field interno + preset dropdown UX**: el path queda como mecánica de persistencia (el `.moodmap` guarda paths que los devs pueden inspeccionar y versionar), pero la UI normal va por preset. Best of both worlds. (4) **Reset per-section sin undo en v1**: trade-off de simplicidad. El dev sabe que reseteó manualmente. Snapshot multi-campo es complejidad innecesaria mientras la operación sea claramente intencional. (5) **Post-Process wrapper sin TreeNode anidado nativo**: ImGui no anida `CollapsingHeader` naturalmente con doble chevron; usar `Indent/Unindent` da el efecto visual sin duplicar el control de colapso. (6) **5 botones reset con IDs únicos por sufijo `##envreset_<id>`**: convención ImGui — labels visibles iguales necesitan IDs distintos. **Suite 926/9705 verde** (sin tests nuevos: Color Grading requiere GL context como BloomPass/SSAOPass; serialización roundtrip cubierta por el patrón existente de EnvironmentComponent + el path es un string adicional ya soportado). Plan archivado en [`archive/plans/PLAN_HITO_F2H58.md`](archive/plans/PLAN_HITO_F2H58.md). **Bug pre-existente detectado y arreglado durante el tour (NO scope F2H58 estrictamente, fix incluido por incompatibilidad)**: F2H57 Bloque E borró un `EndMenu()` del Help menu junto con el submenu Demos — el editor crasheaba al primer hover sobre menus en sesiones reales (no se había manifestado antes porque el test del propio F2H57 no hizo click sobre Help). Pedido del dev al cierre: **F2H59 = "Primitivas Plano + clásicas"** — actualmente el menú Brush>Añadir tiene Box/Cylinder/Sphere/Pyramid/Wedge/Prism (triangular y hexagonal) pero **no Plano**. El dev usaba Box aplastado como suelo y notó la fricción durante el tour: *"la base es un cuadrado comprimido, no tenemos la primitiva de un plano? deberiamos tener… estaria bien que tengamos las primitivas clasicas para no modelar todo desde la base"*. F2H59 abierto post-cierre.
