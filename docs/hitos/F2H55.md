Bloques A-F, primer hito de **Sub-fase 2.6 — Render polish**. Sub-fase 2.5 (Diálogos / Inventario / Quests) cerró en F2H53; F2H54 quedó skip (laptop-only, descartado tras divergencia con desktop). **Origen**: post-cierre F2H53 el dev pidió encarar "lo visual que podemos sacar bien" — entre 4 opciones de polish (bloom / AO / color grading / god rays) eligió bloom como primero por mayor impacto visual inmediato y scope acotado. **Filosofía**: polish no-destructivo + engine-grade. Bloom es post-process pass insertado entre el HDR scene FB y el PostProcessPass existente — si se apaga (intensity=0 o disabled), la escena se ve idéntica a pre-F2H55 (cero regresión). Settings van al `EnvironmentComponent` existente (no se inventa struct nuevo) y se persisten aditivamente al `.moodmap` (back-compat: mapas pre-F2H55 cargan con defaults razonables). **Bloque A — shaders portados de Godot 4 (MIT) + Filament (Apache 2.0)**, algoritmo COD Advanced Warfare 2014 (Sledgehammer/Activision, "Next Generation Post Processing" presentation, estándar industria 10+ años). 4 archivos GLSL nuevos en `shaders/`: `bloom.vert` (fullscreen triangle compartido, mismo patrón que post_process.vert), `bloom_downsample.frag` (13-tap filter con Karis weighted average + soft-knee threshold para evitar fireflies, solo en primer mip; 4 inner taps + 4 outer 2x2 boxes), `bloom_upsample.frag` (tent 9-tap 1-2-1/2-4-2/1-2-1 escalado por radius), `bloom_composite.frag` (`mix(hdr, hdr+bloom, intensity)`). **Bloque B — `BloomPass` C++ class** (`src/systems/render/BloomPass.{h,cpp}`, ~250 líneas). Mip chain de 6 niveles HDR (RGBA16F) recreado on resize. `apply(src, dst, threshold, intensity, radius) → bool` retorna false si size inválido (caller usa src directo para post-process). Flow interno: downsample chain (src → mip[0] con threshold+Karis, mip[0..4] → mip[1..5] sin threshold) → upsample chain con blending aditivo `GL_ONE/GL_ONE` (mip[5..1] → mip[4..0], cada upsample suma al downsample value preexistente) → composite final. Save/restore GL state (depth/cull/blend). Cost ~10MB FBs + ~1ms en GPUs modernas a 1080p. **Wireup en `SceneRenderer`**: `m_bloomPass` + `m_bloomFb` (HDR intermedio) members; `endFrame()` corre bloom antes del PostProcessPass si `bloomEnabled && intensity > 0 && apply() succeeded`; resize del bloomFb cableado junto al sceneFb en `renderScene`. CMakeLists agrega el .cpp. **Bug crítico durante tour** (fixeado): primera versión chequeaba `if (m_mips[0] == nullptr) return;` ANTES del resize interno → mip chain nunca se creaba → apply early-return + endFrame ya había apuntado postProcessSrc a m_bloomFb vacío → pantalla negra. Fix: resize idempotente PRIMERO, retorno bool indicando si escribió al dst, endFrame solo cambia postProcessSrc si apply() retornó true. **Bloque C — `EnvironmentComponent` extendido** con 4 campos: `bloomEnabled` (master switch, off salta el pass entero sin costo), `bloomThreshold` (luminance HDR > threshold contribuye, default 1.0), `bloomIntensity` (peso del bloom sobre HDR, default 0.6), `bloomRadius` (escala del tent filter, default 1.0). `SavedEnvironment` (engine/scene/serialization/SceneSerializer.h) extiende paralelo. `EntitySerializer.cpp` write aditivo (solo persiste si difiere del default — mapas pre-F2H55 round-trip sin ensuciarse) + parse con defaults explícitos (back-compat). `SceneLoader.cpp` copia Saved → Component. `SceneRenderer::applyEnvironmentFromScene` (que corre cada frame en renderScene) copia Component → cache del renderer — esto da la propagación VIVA de sliders sin re-cargar mapa. **Bloque D — UI panel sliders en `InspectorPanel_Light.cpp::renderEnvironmentSection`**: sección "Bloom (glow)" debajo de Post-procesado con checkbox `activar bloom` + 3 sliders (`umbral` 0-3, `intensidad` 0-2, `radio del halo` 0.5-3). Sliders se ocultan si el checkbox está apagado. `pushEditIfDone` para undo records al soltar slider. 5 i18n keys nuevas en es.json + en.json (`environment.bloom*`). **Bloque E — validación visual**: dev creó una caja de testing improvisada en lugar de la demo planeada — confirmó bloom visible (cita: *"el bloom va bien"*). Demo `.moodmap` formal con cubos emissive diferida (no agregaba valor extra al tour). **Bloque F — cierre**. **NO entra en F2H55**: AO de rincones (F2H56 candidate), color grading LUT-based, rayos volumétricos/god rays, lens flares, auto-exposure adaptativo, per-camera environment override. **Decisiones**: (1) **No-librería externa**: bloom no tiene "lib plug-and-play" en la industria — Unreal/Unity/Godot/id/Frostbite todos lo implementan inline porque debe estar pegado al pipeline del motor. Portar shaders de Godot/Filament (open-source comprobado) es el equivalente a "no reinventar" en este dominio. (2) **Settings per-mapa, no global**: cada `.moodmap` guarda su look propio en el `EnvironmentComponent` — sienta la base donde se acumularán futuras opciones de Sub-fase 2.6 (ssao, colorGrading, godRays). (3) **Default bloom on** (no off) — engine-grade no toca semánticas de gameplay pero SÍ provee defaults visuales sensatos; el dev del juego apaga si quiere look plano vintage. (4) **Defaults aditivos en JSON**: solo se escriben campos que difieren del default — los `.moodmap` viejos se mantienen limpios al round-tripear. Suite **926/9705 verde** (sin tests nuevos en F2H55: bloom es shader visual no headless-testeable; serialización roundtrip cubierta por el patrón existente de EnvironmentComponent que ya tenía coverage F2H15).
